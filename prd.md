# 录音文件识别与分析应用 PRD

版本：v0.4  
日期：2026-05-07  
阶段：MVP 需求确认版  
产品定位：面向咨询顾问的 Railway 部署录音访谈分析工作台，后续演进为项目知识问答与企业知识库系统

---

## 1. 背景与目标

咨询顾问和客户经理在项目过程中会产生大量客户访谈、专家访谈、内部访谈录音。单份录音通常为几十分钟到 3 小时，传统整理方式耗时长、可追溯性差，后续写报告或做需求分析时难以快速找到客户原话和证据。

本产品希望通过阿里云语音识别能力和大语言模型能力，将录音文件转化为：

1. 可阅读的 AI 清洁稿。
2. 准确完整的访谈纪要。
3. 可引用、可追溯、可用于报告和需求分析的证据。
4. 可通过自然语言提问调用的项目内访谈材料。

MVP 不追求完整企业级知识库和组织权限体系，而是先打通从录音上传到分析复用的核心闭环。

## 2. 最新需求澄清结论

### 2.1 产品阶段定位

产品整体分三步：

```text
阶段 1：录音文件转写和分析
阶段 2：项目内知识问答应用
阶段 3：企业知识库管理和使用系统
```

MVP 属于阶段 1，但需要为阶段 2 和阶段 3 预留数据结构、日志字段和扩展能力。

### 2.2 MVP 使用规模

| 项目 | 结论 |
|---|---|
| 使用方式 | 先由少数人在本机使用，所有操作默认一个用户 |
| 登录与用户管理 | MVP 不做登录、不做用户管理、不做群组、不做部门维护 |
| 项目数量 | 预计 100 个项目 |
| 单项目录音数 | 平均 10 份，上限 30 份 |
| 累计录音量 | 预计累计 1000 小时，不是一次性导入 |
| 单份录音时长 | 平均 1 小时，最大 3 小时 |
| 单文件大小 | 最大 500M |
| 支持格式 | mp3、wav、m4a、aac、flac、ogg、wma |
| 视频处理 | MVP 不做 |

### 2.3 模型与部署

| 模块 | 结论 |
|---|---|
| ASR | 阿里云 Fun-ASR / 阿里云录音文件识别能力 |
| LLM | 阿里云通义/百炼大语言模型 API |
| 对象存储 | Railway Storage Buckets |
| 应用部署 | MVP 阶段部署在 Railway |
| 数据库 | Railway PostgreSQL |
| 任务队列 | Railway Redis |
| Worker | Railway Worker 服务，负责 ASR/LLM/导出等异步任务 |
| 后续部署 | Railway 跑通 MVP 后，再评估国内云、私有化或企业服务器部署 |
| 上传方式 | MVP 当前采用后端代理上传：先创建录音记录，再由后端写入 Railway Storage Bucket；后续可评估直传 |
| ASR 结果获取 | MVP 优先轮询；Railway 具备公网地址，后续可评估回调 |

### 2.4 核心功能边界

MVP 必须做：

- 首页：项目列表、新建项目、按标题检索项目。
- 项目页：三栏布局，左侧录音文件，中间清洁稿和播放器，右侧纪要/问答。
- 音频上传：支持主流音频格式，最大 500M。
- 自动转写：上传后进入转写任务。
- 自动生成清洁稿。
- 自动生成访谈纪要。
- 清洁稿轻量编辑和保存。
- 内置 2 个纪要模板：客户访谈纪要、专家/内部访谈纪要。
- 自然语言项目问答：用户最多勾选 10 份录音，把清洁稿作为上下文材料回答。
- 如果上下文超过模型上限，提示减少文件数量，MVP 不自动摘要。
- 导出纪要和文本材料。
- 处理队列与失败重试。
- 项目级成本与任务级成本统计。
- 日志字段和诊断包预留。

MVP 不做：

- 复杂登录、用户管理、群组、部门、细粒度权限（MVP 仅保留固定 admin 登录）。
- RAG、向量数据库、企业知识库。
- 术语库、热词库。
- 自定义模板和模板管理。
- 视频处理。
- 企微 SSO 或组织同步。
- AI 日志自动分析。
- 复杂后台管理。

## 3. 用户画像

### 3.1 核心用户：咨询顾问

使用场景：

- 上传客户访谈录音。
- 快速获得清洁稿和访谈纪要。
- 在报告撰写、需求分析、规划方案中引用客户原话。
- 对同一项目的多份访谈进行提问。

核心诉求：

- 不想花大量时间手工整理录音。
- 访谈纪要要准确完整。
- AI 结论必须能回到原文和时间戳。
- 项目问答可以跨多份录音找材料。

### 3.2 次要用户：客户经理

使用场景：

- 上传客户沟通录音。
- 快速提取客户需求、关注点、异议和后续动作。
- 生成客户跟进材料。

### 3.3 后续用户：PMO / 开发人员

MVP 阶段只使用系统设置、任务监控、成本用量和诊断包。

使用场景：

- 查看任务失败原因。
- 统计项目处理成本。
- 导出诊断包辅助排查 bug。
- 为后续功能优化分析预留日志数据。

## 5. 核心业务流程

### 5.1 用户主流程

```text
进入首页
  -> 新建项目或搜索已有项目
  -> 进入项目页
  -> 上传录音文件
  -> 创建上传会话和 uploading 录音记录
  -> 前端通过后端上传文件内容
  -> 后端写入 Railway Storage Bucket
  -> 创建处理任务
  -> 调用阿里 ASR
  -> 保存原始转写稿
  -> 生成 AI 清洁稿
  -> 自动生成访谈纪要
  -> 用户查看、轻量编辑、导出
  -> 用户勾选最多 10 份录音进行项目问答
```

### 5.2 任务状态流

```text
created
  -> uploading
  -> queued
  -> asr_processing
  -> asr_completed
  -> cleaning
  -> cleaning_completed
  -> summary_generating
  -> completed

任意处理状态 -> failed
失败任务 -> 从失败节点重试并覆盖该节点结果
```

用户侧简化展示：

```text
上传中 / 排队中 / 转写中 / 生成清洁稿 / 生成纪要 / 已完成 / 失败
```

### 5.3 两层文本逻辑

MVP 只保存两层文本：

1. 原始转写稿：ASR 输出，只读保存，用于追溯。
2. AI 清洁稿：LLM 对原始稿进行标点、分段、口语清理和轻度纠错后的版本。

交互规则：

- 默认展示 AI 清洁稿。
- 用户可点击显示原始稿。
- 原始稿置灰，只读展示。
- 清洁稿支持轻量编辑和保存。
- 如果清洁稿被编辑，纪要标记为“可能需要重新生成”。
- 用户可以在纪要过期提示旁手动重新生成纪要；不在顶部固定展示重新生成按钮。

### 5.4 纪要生成逻辑

转写完成后自动生成纪要。

内置模板：

1. 客户访谈纪要。
2. 专家/内部访谈纪要。

客户访谈纪要建议结构：

```text
1. 访谈背景
2. 受访者角色与关注点
3. 核心结论
4. 关键痛点
5. 需求分析
6. 风险与顾虑
7. 后续待办
8. 可引用原话
```

专家/内部访谈纪要建议结构：

```text
1. 访谈背景
2. 核心观点
3. 行业/业务判断
4. 关键事实
5. 分歧或不确定点
6. 对项目的启发
7. 后续追问建议
8. 可引用原话
```

### 5.5 项目问答逻辑

MVP 不做 RAG，采用直接上下文方案。

规则：

- 左侧文件列表支持勾选录音，最多勾选 10 份。
- 进入项目后默认勾选前 10 份录音；如果项目超过 10 份，用户发送问题时提示本次仅使用这 10 份，并建议按问题精准勾选。
- 连续对话默认沿用上一轮勾选文件；切换历史对话时恢复该对话上一轮使用的文件范围。
- 问答时将所选录音的 AI 清洁稿作为材料上下文，并自动带入最近 4 轮用户/AI 问答作为对话上下文。
- AI 回答应尽量带来源文件名和时间戳。
- 如果所选材料超过模型上下文上限：
  - 优先提示用户减少文件数量。
  - MVP 不提供自动摘要压缩。

处理方式：

```text
用户问题 + 已选录音清洁稿
  -> 上下文长度检查
  -> 未超限：直接调用长上下文模型回答
  -> 超限：提示减少文件
  -> 返回答案、来源文件、时间戳
```

- 左侧录音文件名可通过铅笔图标行内修改，回车保存；只修改展示名，不修改对象存储 key。

### 8.2 技术选型建议

推荐技术栈：

| 层级 | 选型 |
|---|---|
| 前端 | React + TypeScript + Vite |
| UI 组件 | Ant Design |
| 后端 | Python FastAPI |
| 数据库 | PostgreSQL |
| 任务队列 | Redis + RQ 或 Celery |
| 文件存储 | Railway Storage Buckets |
| ASR | 阿里云 Fun-ASR |
| LLM | 阿里云通义/百炼模型 API |
| 音频处理 | ffmpeg，可选 |
| 部署 | Railway Web 服务 + Railway Worker 服务 |

给产品经理看的选择理由：

- React 适合做三栏工作台、独立滚动、播放器联动和复杂交互。
- Ant Design 适合快速搭建企业工具界面。
- FastAPI 适合做 AI 应用后端，接模型 API、处理文件、调任务队列都方便。
- Railway PostgreSQL 足够支撑 MVP 的项目、录音、纪要、任务、日志和成本数据。
- Railway Redis 队列用于处理长录音和模型任务，避免页面卡死。
- Railway Storage Buckets 提供 S3 兼容对象存储，适合保存录音和导出文件。
- MVP 不做 RAG，所以暂不引入向量数据库，降低复杂度。
- Railway 可以同时托管 Web/API、Worker、PostgreSQL、Redis 和对象存储，MVP 架构更集中。

### 8.3 文件上传与转写流程

```text
用户选择音频文件
  -> 前端请求后端生成 Railway Bucket 预签名上传 URL
  -> 前端直传 Railway Storage Bucket
  -> 上传完成后通知后端
  -> 后端创建 recording 和 processing_job
  -> Worker 生成音频预签名下载 URL
  -> Worker 调用阿里 ASR 创建识别任务
  -> Worker 轮询识别结果
  -> 保存原始转写稿
  -> 调用 LLM 生成清洁稿
  -> 调用 LLM 生成纪要
  -> 更新任务状态为已完成
```

## 9. 风险与应对

| 风险 | 影响 | MVP 应对 |
|---|---|---|
| 500M 大文件上传失败 | 用户无法处理长音频 | MVP 使用后端代理上传到 Railway Bucket，前端显示进度；上传完成后立即进入列表并异步处理 |
| ASR 回调链路不稳定 | 任务无法完成 | MVP 优先使用轮询方式获取 ASR 结果，后续再评估回调 |
| 主流音频格式兼容不稳定 | ASR 失败 | 支持上传多格式，必要时用 ffmpeg 转码 |
| 10 份录音上下文过长 | 项目问答失败或成本高 | 检查上下文长度，提示减少文件 |
| LLM 纪要不准确 | 影响报告和需求分析 | 保留原始稿和清洁稿，支持重新生成 |
| 清洁稿被编辑后纪要过期 | 用户引用旧结果 | 标记纪要可能需要重新生成 |
| 没有用户系统导致溯源弱 | 后续多人使用困难 | 预留 user_id，MVP 默认 default_user |
| 没有部门系统导致分摊弱 | 无法部门成本统计 | MVP 先做项目/任务成本，预留 department_id |
| 日志保存不足 | 后续 bug 难排查 | MVP 预留操作、任务、模型、错误、性能日志 |

## 10. MVP 验收标准

### 10.1 业务验收

- 可以创建项目、搜索项目、进入项目。
- 可以在项目中上传音频文件。
- 支持 mp3、wav、m4a、aac、flac、ogg、wma。
- 单文件最大支持 500M。
- 上传后自动创建处理任务。
- 转写完成后自动生成清洁稿。
- 清洁稿可查看、轻量编辑和保存。
- 转写完成后自动生成纪要。
- 纪要支持手动重新生成。
- 可以导出清洁稿和纪要。
- 可以勾选最多 10 份录音进行项目问答。
- 问答结果尽量返回来源文件和时间戳。
- 超出上下文时能提示减少文件。

### 10.2 系统验收

- 文件直接上传 Railway Storage Bucket。
- 后端不转发 500M 大文件。
- ASR 调用阿里云接口。
- ASR 结果通过轮询获得，后续可评估回调。
- LLM 调用阿里云模型。
- 任务状态可追踪。
- 失败任务可查看原因和重试。
- 项目级和任务级成本可查看。
- 诊断包可导出。

### 10.3 数据验收

- 每个项目记录录音数量、总时长、最近更新时间。

### 12.6 任务与重试规则

处理链路：

```text
upload_complete
  -> asr_transcription job
  -> clean_transcript job
  -> summary_generation job
  -> completed
```

规则：

- 每个阶段是独立 job。
- 哪个阶段失败，页面就显示哪个阶段失败。
- 只有失败阶段提供重试按钮。
- 重试从失败阶段重新执行，成功后覆盖该阶段结果。
- ASR 或清洁稿成功后，不提供“重试成功节点”的常规入口。
- 用户手动编辑清洁稿后，不自动重新生成纪要；用户可点击重新生成纪要。
- 上传失败保留草稿状态，用户可以删除或重新上传。
- ASR 失败后保留文件。
- 项目问答上下文超限时，不自动重试；用户修改勾选文件后才能重新发起。
- MVP 不做 token 估算和摘要压缩。

后续上下文控制建议：

```text
先通过代码统计每个文件字数；
累计字数超过 10 万字时；
对超过 2 万字的单文件分别做摘要。
```

## 13. 2026-05-07 交互与对话机制更新

### 13.1 首页最终布局

首页采用轻量表格布局，不使用卡片网格：

```text
录音分析工作台                                      设置   退出

[ 搜索项目...... ]                         [ + 新建项目 ]

项目                         文件数量    总时长(h)    最近更新
XX项目                       12          11.1         2026-05-07   ...
XX项目                       1           1.1          2026-05-07   ...
```

交互规则：

- 搜索框和 `+ 新建项目` 在同一行。
- 点击 `+ 新建项目` 后弹窗输入项目名称。
- 确认创建后直接进入项目页。
- 项目一行一个，点击项目行任意非菜单区域进入项目。
- 最右侧 `...` 菜单只包含：修改项目名称、删除项目。
- 删除项目二次确认。
- 首页不展示最近处理任务。
- `录音数量` 统一改为 `文件数量`。
- `总时长(h)` 以小时展示，保留 1 位小数。
- `最近更新` 显示到 `YYYY-MM-DD`。
- 默认排序按更新时间倒序。

### 13.2 原始稿展示

- 默认显示 AI 清洁稿。
- 点击“显示原始稿”后，在每个清洁稿段落下方置灰展示原始稿。
- 原始稿不加“原始稿：”前缀。

### 13.3 纪要 Markdown 化

纪要不再由前端代码固定排版和字段。LLM 直接输出 Markdown，后端保存 Markdown，前端只负责 Markdown 渲染。

规则：

- 纪要 Prompt 控制章节结构。
- 前端不写死“访谈背景、核心结论、痛点”等字段。
- 导出纪要时直接导出同一份 Markdown。
- 清洁稿导出仍由系统按段落拼接为 Markdown。

建议纪要包含：访谈摘要、关键结论、客户痛点、需求分析、风险与顾虑、后续建议、报告引用建议。

### 13.4 多轮问答机制

问答改为“对话线程 + 多轮消息”。

上下文分为两部分：

1. 当前用户勾选文件的清洁稿。用户可以随时修改勾选文件，每次发送问题时读取发送瞬间的勾选文件。
2. 当前对话的问答上下文。默认上限 5 轮，每次调用模型时自动带入前 4 轮用户和 AI 问答。

每次发起对话时，后端将上述两部分稍作封装后作为 Prompt 提交给 LLM API。

规则：

- 支持新建对话。
- 支持历史对话下拉选择。
- 历史对话列表展示“该对话第一个问题前 10 个字 + 最后一次对话时间”。
- 时间格式为 `MM-DD HH:MM`，例如 `05-07 12:03`。
- 每一轮回答保存本轮使用的文件集合，便于用户追溯答案依据。
- 当前勾选为空时不允许发送。
- 当前勾选超过 10 份时不允许发送。
- 当前勾选清洁稿累计超过 10 万字时提示减少文件，MVP 不自动摘要。

## 15. 2026-05-08 Railway 可运行版本补充

### 15.1 部署形态

MVP 当前以 Railway 为目标部署环境：

1. Web 服务：FastAPI 后端托管 React/Vite 构建后的静态前端，同时对外提供业务 API。
2. Worker 服务：独立 RQ Worker，消费 Redis 队列中的 ASR、清洁稿、纪要、问答任务。
3. 数据库：Railway PostgreSQL。代码兼容 Railway 常见 `postgres://` / `postgresql://` 连接串。
4. 队列：Railway Redis。
5. 文件：Railway Storage Bucket，通过 S3 兼容预签名 URL 上传和下载。
6. 模型：阿里云 DashScope / 百炼 ASR 与 OpenAI 兼容 LLM 接口。

### 15.2 真实 ASR 编排

录音上传完成后，系统按以下节点执行，每个节点为独立 Job：

```text
上传完成 -> ASR 转写 -> 清洁稿生成 -> 纪要生成
```

ASR 节点不再使用占位逻辑，真实流程为：

1. Web 生成录音文件的 Bucket 下载预签名 URL。
2. Worker 调用阿里云录音文件识别提交任务接口，传入 `file_urls`。
3. Worker 保存外部 `task_id` 到 `processing_jobs.external_task_id`。
4. Worker 轮询阿里云任务查询接口，直到 `SUCCEEDED`、`FAILED` 或超时。
5. 任务成功后下载 `transcription_url` 对应 JSON。
6. 解析 `transcripts[].sentences[]` 为原始转写段落，保留 `speaker_id`、`begin_time`、`end_time`、`text`。
7. ASR 失败只标记 ASR 节点失败，用户可从该节点重试。

### 15.3 Railway 上线前必填配置

Railway Web 与 Worker 两个服务需要共享同一套环境变量：

```text
APP_ENV=production
APP_BASE_URL=https://<Railway Web 域名>
SESSION_SECRET=<随机字符串>
ADMIN_USERNAME=admin
ADMIN_PASSWORD=mp2026
DATABASE_URL=<Railway PostgreSQL URL>
REDIS_URL=<Railway Redis URL>
QUEUE_SYNC=false
STORAGE_PROVIDER=railway_bucket
STORAGE_MOCK_ENABLED=false
ASR_MOCK_ENABLED=false
LLM_MOCK_ENABLED=false
```

AI 与 Bucket 配置既可以通过环境变量初始化，也可以在系统设置页写入数据库；写入后 Web 与 Worker 会通过同一个 PostgreSQL 读取配置。

### 15.4 仍需人工确认的运行项

1. 阿里云 ASR 模型名保持可配置。官方 Paraformer REST 示例使用 `paraformer-v2`；如 `fun-asr` 在当前账号不可用，需要在系统设置中改为实际开通的模型名。
2. Railway Bucket 必须能生成阿里云 ASR 可访问的 HTTPS 预签名下载 URL。
3. 3 小时录音不默认开启说话人分离，避免触发长音频识别失败或超时风险；说话人仍可在清洁稿中编辑。
